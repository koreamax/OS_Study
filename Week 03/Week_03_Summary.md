# Chapter 03. 명령어

컴퓨터가 이해하는 두 정보 중 **명령어**를 다룬다. 먼저 개발자가 작성한 소스 코드가 어떻게 명령어로 변환되는지 살펴보고, 이어서 명령어 하나하나의 구조를 알아본다.

## 03-1 소스 코드와 명령어

### 고급 언어와 저급 언어

개발자가 작성한 소스 코드를 컴퓨터가 그대로 직독직해하지는 못한다. 언어를 두 종류로 나누어 이해해야 한다.

| 구분 | 설명 | 예시 |
| --- | --- | --- |
| **고급 언어** | 개발자가 읽고 쓰기 편하게 만들어진 언어 | C, C++, Java, Python 등 |
| **저급 언어** | 컴퓨터가 이해하고 실행하기 위해 만들어진 언어 (= 명령어) | 기계어, 어셈블리어 |

고급 언어로 작성된 소스 코드는 컴퓨터 내부에서 실행될 때 **저급 언어로 변환**되어 실행된다.

#### 저급 언어의 두 종류

**기계어**
- 0과 1(이진수)로 표현된, 컴퓨터가 직접 이해하는 명령어다.
- 표기할 때 이진수로 나타내기도 하고 십육진수로 나타내기도 한다.
- 사람이 읽기에는 매우 어렵다. 컴퓨터를 위해 만들어진 언어이기 때문이다.

**어셈블리어**
- 0과 1로 이루어진 기계어를 사람이 읽고 쓰기 편한 형태로 번역한 저급 언어다.
- 한 줄 한 줄이 곧 컴퓨터를 동작시키는 명령어다.
- 기계어와 달리 소스 코드에 직접 명시하며 개발하기도 하므로, 알아두면 도움이 된다.

### 컴파일 언어와 인터프리터 언어

고급 언어가 저급 언어로 변환되는 방식에는 크게 **컴파일 방식**과 **인터프리터 방식**이 있다.

#### 컴파일 언어

컴파일러라는 특수한 프로그램에 의해 소스 코드 **전체**가 한꺼번에 저급 언어로 변환되는 언어다.

- 소스 코드가 컴파일러에 의해 변환되는 과정을 **컴파일**이라 하고, 그 결과로 생성된 저급 언어를 **목적 코드**(오브젝트 코드)라고 한다.
- 컴파일러는 소스 코드를 처음부터 끝까지 훑으며 오류, 사용되지 않는 변수, 최적화 가능 여부 등을 따진 뒤 전체를 통째로 변환한다.
- 소스 코드 중간에 오류가 하나라도 있으면 **컴파일 에러**가 발생해 변환 자체가 되지 않는다.
- 대표 언어: C, C++

#### 인터프리터 언어

인터프리터라는 특수한 프로그램에 의해 소스 코드가 **한 줄씩** 실행되는 언어다.

- 한 줄씩 변환·실행하므로 소스 코드 전체가 변환되기를 기다릴 필요가 없다.
- 소스 코드 중간에 오류가 있어도, 그 직전까지의 코드가 올바르면 거기까지는 정상 실행된다.
- 대표 언어: Python (실행 시 **바이트 코드**라는 중간 단계 코드를 생성), Java 등

> **비유**
> - **컴파일 방식**: 독일어를 모르는 친구에게 독일어 책 전체를 한국어로 번역해 전달하는 것.
> - **인터프리터 방식**: 독일어를 모르는 친구 곁에서 책을 한 줄씩 읽어 주는 것.

> **모든 언어가 둘로 나뉘는가?**
> 그렇지 않다. 두 방식의 경계가 애매한 경우, 둘 다 가능한 경우도 있다. 컴파일·인터프리터 방식은 고급 언어가 저급 언어로 변환되는 **대표적인 방식**으로 이해하는 것이 좋다.

### 목적 파일 vs 실행 파일

컴파일 결과로 생성되는 코드를 **목적 코드**, 이를 담은 파일을 **목적 파일**이라 한다. 한편 우리가 실제 실행하는 **실행 파일**을 이루는 코드는 **실행 코드**다.

목적 파일과 실행 파일은 **둘 다 기계어로 이루어져 있지만 서로 다르다.** 목적 파일은 **링킹**이라는 과정을 거친 이후에야 실행 파일이 된다.

#### C 언어의 컴파일 과정

소스 코드가 실행 파일이 되기까지는 사전적 정의(소스 코드 → 컴파일 → 목적 코드)보다 더 자세한 단계가 있다. C 언어의 경우 다음 네 단계를 거친다. (보통 컴파일러가 이 과정을 한 번에 처리하기 때문에 내부 단계를 놓치기 쉽다.)

```
소스 코드(.c)
   │  전처리기(preprocessor)
   ▼
전처리된 소스 코드(.i)
   │  컴파일러(compiler)
   ▼
어셈블리어(.s)
   │  어셈블러(assembler)
   ▼
목적 파일(.o)  ← 목적 코드(기계어)
   │  링커(linker)
   ▼
실행 파일
```

| 단계 | 담당 | 입력 → 출력 | 설명 |
| --- | --- | --- | --- |
| **전처리(preprocessing)** | 전처리기 | `.c` → `.i` | 컴파일 전 준비 작업 |
| **컴파일(compiling)** | 컴파일러 | `.i` → `.s` | 전처리된 소스 코드를 어셈블리어로 변환 |
| **어셈블(assembling)** | 어셈블러 | `.s` → `.o` | 어셈블리어를 기계어(목적 코드)로 변환 |
| **링킹(linking)** | 링커 | `.o` → 실행 파일 | 여러 목적 파일을 하나의 실행 파일로 연결 |

**1. 전처리(preprocessing)**

본격적인 컴파일 전에 준비 작업을 하는 단계다. 주요 작업은 다음과 같다.

- 외부에 선언된 소스 코드(라이브러리)를 포함시키기 — 예) `#include <stdio.h>`를 만나면 `stdio.h`의 모든 내용을 소스 코드 안으로 그대로 가져온다. (`printf` 함수를 쓰려면 `stdio.h`가 필요하다.)
- 편의를 위해 작성된 매크로를 변환하기
- 컴파일할 영역 명시하기

전처리가 끝나도 여전히 C 언어로 된 소스 코드다.

**2. 컴파일(compiling)**

전처리된 소스 코드를 컴퓨터가 이해할 수 있는 저급 언어, 즉 **어셈블리어**로 변환한다. (어셈블리어의 모습은 CPU와 컴파일러의 종류에 따라 달라질 수 있다.)

**3. 어셈블(assembling)**

어셈블리어를 **기계어(0과 1)**로 변환한다. 그 결과로 목적 코드를 담은 **목적 파일**이 만들어진다.

**4. 링킹(linking)**

각기 다른 목적 파일을 하나의 실행 파일로 연결(묶음)하는 작업이다.

> **링킹이 필요한 이유** — `main.c`와 `helper.c`를 각각 목적 코드로 만들었다고 하자. 둘 다 기계어이지만, `main.c`가 `helper.c`의 기능(예: `helper_add`)을 사용한다면 `main.o` 하나만으로는 실행되지 않는다. 두 목적 파일을 연결해 줘야 비로소 실행 파일이 된다. 이 연결 작업이 링킹이다.

> 이 단계는 컴파일 언어를 사용하는 개발자라면 더 자세히 알아두는 것이 도움이 된다.

## 03-2 명령어의 구조

### 연산 코드와 오퍼랜드

명령어는 사람이 명령을 내릴 때처럼 **"무엇을 대상으로 무엇을 수행하라"**는 형식을 따른다. 일반적으로 명령어는 **연산 코드**와 **오퍼랜드**로 이루어진다.

| 구성 요소 | 의미 |
| --- | --- |
| **연산 코드(operation code)** | 수행할 연산 |
| **오퍼랜드(operand)** | 연산에 사용될 데이터, 또는 그 데이터가 **저장된 위치** |

> 오퍼랜드 필드에는 데이터 자체보다 **데이터가 저장된 위치**가 훨씬 더 자주 담긴다. 그래서 오퍼랜드 필드를 **주소 필드**라고 부르기도 한다.

기계어, 어셈블리어도 결국 명령어이므로 연산 코드와 오퍼랜드로 이루어져 있다.

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/ab4f0f4a-82fc-4356-99ea-56777ca36939" />


**오퍼랜드의 개수** — 명령어에 따라 오퍼랜드는 0개일 수도, 1개 이상일 수도 있다.

| 오퍼랜드 개수 | 명칭 |
| --- | --- |
| 0개 | 0-주소 명령어 |
| 1개 | 1-주소 명령어 |
| 2개 | 2-주소 명령어 |
| 3개 | 3-주소 명령어 |

#### 대표적인 연산 코드의 종류

연산 코드는 CPU마다 종류와 개수가 다르다. 대부분의 전공서가 공통으로 기술하는 유형은 크게 네 가지다. (개별 명령어를 외우기보다 유형을 파악하는 것이 중요하다.)

| 유형 | 대표 명령어 | 설명 |
| --- | --- | --- |
| **데이터 전송** | MOVE, STORE, LOAD(FETCH), PUSH, POP | 데이터를 옮기거나 저장·적재한다 |
| **산술/논리 연산** | ADD, SUBTRACT, MULTIPLY, DIVIDE, AND, OR, NOT, COMPARE | 사칙연산, 논리 연산, 값 비교 |
| **제어 흐름 변경** | JUMP, CONDITIONAL JUMP, HALT, CALL, RETURN | 실행 순서를 특정 주소로 옮기거나 멈춤, 함수 호출·복귀 |
| **입출력 제어** | READ(INPUT), WRITE(OUTPUT), START IO, TEST IO | 입출력장치에 읽기/쓰기/시작/상태 확인 |

> **CALL / RETURN** — 함수를 다뤄 본 적이 있다면 이해하기 쉽다. CALL은 되돌아올 주소(리턴 주소)를 저장한 채 특정 주소(함수)를 실행하라는 명령이고, 함수 실행이 끝나면 RETURN으로 호출 지점으로 돌아온다.

### 주소 지정 방식

오퍼랜드 필드에는 데이터 자체를 직접 쓰지 않고 데이터가 저장된 위치를 쓰는 경우가 많다. 그 이유는 **명령어 내에서 표현할 수 있는 데이터의 크기가 제한되기 때문**이다.

> **예시** — 명령어 크기가 16비트이고 연산 코드에 4비트, 오퍼랜드 2개에 각 6비트가 할당된다면, 하나의 오퍼랜드로 표현 가능한 데이터는 2⁶ = 64가지뿐이다. 하지만 오퍼랜드에 "메모리 주소"를 적고 그 주소에 16비트 데이터를 저장하면 2¹⁶가지를 표현할 수 있다.

**유효 주소(effective address)** — 연산에 사용될 데이터가 **실제로 저장된 위치**를 말한다.

연산에 사용될 데이터의 위치(= 유효 주소)를 찾는 방법을 **명령어 주소 지정 방식**이라고 한다. 대표적인 방식은 다음과 같다.

| 방식 | 오퍼랜드 필드에 담기는 것 | 특징 |
| --- | --- | --- |
| **즉시 주소 지정** | 연산에 사용될 **데이터 자체** | 가장 단순하고 빠름. 단, 표현 가능한 데이터 크기가 제한됨 |
| **직접 주소 지정** | 유효 주소(데이터가 있는 **메모리 주소**) | 유효 주소를 바로 찾아감. 표현 가능한 주소 크기가 연산 코드만큼 줄어듦 |
| **간접 주소 지정** | **유효 주소가 저장된 주소** | 유효 주소를 크게 확보 가능. 메모리를 여러 번 접근하므로 가장 느림 |
| **레지스터 주소 지정** | 데이터가 저장된 **레지스터** | 직접 주소 지정과 유사하나, 레지스터 접근이라 더 빠름 |
| **레지스터 간접 주소 지정** | 데이터가 있는 메모리 주소를 담은 **레지스터** | 레지스터를 거쳐 메모리의 데이터에 접근 |

> **중요한 상식** — 레지스터는 CPU 내부에, 메모리는 CPU 외부에 있다. 따라서 **레지스터 접근 속도가 메모리 접근 속도보다 훨씬 빠르다.** 메모리 접근을 최소화하는 것이 속도 면에서 유리하다.

### 스택과 큐

데이터 전송 연산 코드(PUSH/POP 등)와 함께 자주 등장하는 자료구조다. 컴퓨터 구조뿐 아니라 여러 분야에서 매우 자주 언급된다.

**스택(Stack)**
- 한쪽 끝이 막혀 있는 통과 같은 자료구조.
- **나중에 들어간 데이터가 먼저 나온다.** → **LIFO**(Last In First Out)
- 관련 연산: **PUSH**(데이터를 밀어 넣음), **POP**(데이터를 빼냄)

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/a2b424b2-18be-4fc6-b918-3278c9c0aa45" />


**큐(Queue)**
- 양쪽 끝이 뚫려 있는 통과 같은 자료구조.
- **먼저 들어간 데이터가 먼저 나온다.** → **FIFO**(First In First Out)

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/7d3c24ba-b6bd-4ca1-9814-db0144959221" />


> 주소 지정 방식 중에는 특정 레지스터를 학습해야 이해할 수 있는 중요한 방식들도 있으며, 이는 레지스터를 학습한 뒤 다시 다룬다.
