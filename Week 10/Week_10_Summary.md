# Chapter 09. 운영체제 시작하기

컴퓨터 부품들은 전기만 공급하면 마법처럼 동작하는 것이 아니라, **운영체제**라는 가장 근원적인 프로그램에 의해 동작한다. 운영체제 파트의 시작으로 운영체제가 무엇인지, 왜 알아야 하는지, 그리고 전체 그림을 살펴본다.

## 09-1 운영체제를 알아야 하는 이유

### 운영체제란

윈도우, macOS, 리눅스, 안드로이드, iOS 등이 모두 **운영체제(OS, Operating System)**다.

**자원(resource, 시스템 자원)** — 프로그램 실행에 마땅히 필요한 요소. 모든 프로그램은 실행을 위해 자원을 필요로 한다. (컴퓨터의 4가지 핵심 부품이 포함되며, 하드웨어뿐 아니라 네트워크 패킷·파일 등도 자원에 속한다.)

**운영체제**는 이러한 자원을 실행할 프로그램에 **할당**해 주고, 각 프로그램이 올바르게 실행되도록 돕는 특별한 프로그램이다. 운영체제 자신도 프로그램이므로 메모리에 적재되어 실행되는데, 메모리는 두 영역으로 나뉜다.

| 영역 | 적재되는 것 |
| --- | --- |
| **커널 영역(kernel space)** | 운영체제 |
| **사용자 영역(user space)** | 응용 프로그램(메모장, 웹브라우저 등 사용자가 실행하는 일반 프로그램) |

**운영체제의 관리 역할** — 운영체제는 응용 프로그램과 하드웨어 사이에서 자원을 관리한다.

| 관리 대상 | 하는 일 |
| --- | --- |
| **메모리** | 새 프로그램을 빈 공간에 적재하고, 쓰지 않는 프로그램은 삭제해 메모리를 비운다 (사용자가 적재 주소를 지정하지 않아도 됨) |
| **CPU** | 여러 프로그램을 빠르게 번갈아 실행할 때, 어떤 프로그램을 먼저·얼마나 오래 실행할지 결정한다 |
| **입출력장치** | 한 번에 하나만 접근 가능한 장치(예: 프린터)를 여러 프로그램이 충돌 없이 쓰도록 관리한다 |
| **파일** | 보조기억장치의 정보를 파일과 폴더(윈도우)/디렉터리(macOS·리눅스)로 묶어 관리한다 |

> **정부 비유** — 운영체제는 한 나라의 자원(땅·인력·돈)을 국민에게 효율적으로 배분하고 문제를 해결하는 정부와 같다. 정부에 자원별 부서가 있듯, 운영체제도 CPU 관리·프로세스 관리·파일 시스템·메모리 관리 부분으로 나뉘어 있다.

### 운영체제를 알아야 하는 이유

운영체제가 없다면 개발자는 **하드웨어에 직접 접근·조작하는 코드**를 일일이 작성해야 한다. 예를 들어 "1과 2를 더한 결과를 모니터에 출력"하는 단순한 프로그램조차, 운영체제 없이는 메모리 적재·CPU 연산·모니터 출력 코드를 모두 직접 짜야 한다.

- `printf` 같은 단순한 함수도 내부적으로는 **운영체제의 도움을 받아** 실행된다.
- 운영체제 덕분에 개발자는 하드웨어 접근 코드를 직접 작성할 필요 없이, 운영체제에 **"부탁"**(운영체제 코드 실행)하면 된다. (예: 파일 생성·읽기, 화면 출력)

**그렇다면 왜 알아야 하는가?** — 운영체제는 사용자가 아니라 **프로그램을 위한 프로그램**이다. 개발자가 만드는 프로그램은 운영체제의 도움을 받으며 실행되고, 문제가 생기면 하드웨어와 가장 밀접한 운영체제가 가장 먼저 알아챈다. 또한 운영체제는 하드웨어와 달리 오류 메시지 등으로 **개발자와 대화**할 수 있다. 운영체제를 깊이 이해하면 오류 메시지 이해와 문제 해결력이 높아지므로, 컴퓨터 구조를 학습한 뒤 반드시 함께 공부해야 한다.

## 09-2 운영체제의 큰 그림

### 커널

운영체제는 현존하는 프로그램 중 규모가 가장 큰 축에 속하고(리눅스는 소스 코드 수천만 줄), 종류도 다양하다. 그럼에도 운영체제가 제공하는 **핵심 서비스**는 공통적으로 존재한다. 대표적으로 **컴퓨터 자원에 접근·조작하는 기능**과 **프로그램이 올바르고 안전하게 실행되도록 관리하는 기능**이다.

이 핵심 서비스를 담당하는 운영체제의 한 부분(자동차의 엔진, 사람의 심장 같은)을 **커널(kernel)**이라 한다. 운영체제 종류가 달라도 커널은 대동소이하므로, 대부분의 전공서는 운영체제를 설명할 때 커널을 다룬다. (이후 '운영체제'는 특별한 언급이 없으면 커널을 지칭한다.)

> **커널에 속하지 않는 기능 — 유저 인터페이스(UI)** — 사용자와 상호작용하는 통로로, 아이콘·마우스로 조작하는 **GUI**(그래픽 유저 인터페이스)와 명령어로 조작하는 **CLI**(커맨드라인 인터페이스)가 있다. UI는 상호작용 통로일 뿐 운영체제의 핵심 목적은 아니므로 커널에 속하지 않는다. (같은 리눅스 커널이라도 서로 다른 UI를 쓸 수 있다.)

### 이중 모드와 시스템 호출

사용자가 실행하는 일반적인 응용 프로그램은 자원(CPU, 메모리, SSD 등)에 **직접 접근할 수 없다.** 직접 접근이 가능하면 프로그램 간 충돌·데이터 삭제 등 위험이 있기 때문이다. 운영체제는 응용 프로그램이 **오직 자신을 통해서만** 자원에 접근하도록 해 자원을 보호하는 **문지기** 역할을 한다.

#### 이중 모드

**이중 모드(dual mode)**는 CPU가 명령어를 실행하는 모드를 두 가지로 구분하는 방식이다.

| 모드 | 설명 |
| --- | --- |
| **사용자 모드(user mode)** | 운영체제 서비스를 제공받을 수 없는 실행 모드. 자원에 접근하는 명령어를 실행하면 오류 발생 |
| **커널 모드(kernel mode)** | 운영체제 서비스를 제공받을 수 있는 실행 모드. 자원 접근을 포함한 모든 명령어 실행 가능 |

> CPU의 **플래그 레지스터** 안 **슈퍼바이저 플래그**가 현재 커널 모드인지 사용자 모드인지를 나타낸다. CPU는 이 플래그를 보고 자원 접근 가능 여부를 판단한다.

#### 시스템 호출

**시스템 호출(system call)**을 하면 사용자 모드에서 **커널 모드로 전환**되어 운영체제 코드를 실행(운영체제 서비스를 제공받을 수)할 수 있다.

- 시스템 호출은 일종의 **소프트웨어 인터럽트**다. 특정 명령어로 인터럽트를 걸어 커널 모드로 전환한다.
- 처리 방식이 하드웨어 인터럽트와 유사하다: 실행 중 자원 접근이 필요하면 시스템 호출 → 커널 모드로 전환해 운영체제 코드 실행 → 다시 복귀해 사용자 모드로 실행.
- 응용 프로그램은 자원에 빈번하게 접근하므로 시스템 호출도 자주 일어나며, **커널 모드와 사용자 모드를 매우 빈번하게 오간다.**
- 시스템 호출은 운영체제마다 정해져 있다. (예: 리눅스의 `fork`, `exit` 등)

### 운영체제의 핵심 서비스

응용 프로그램은 커널로부터 크게 세 가지 도움을 받으며 실행된다.

#### 1. 프로세스 관리

**프로세스(process)**는 메모리에 적재되어 실행 중인 프로그램이다. (일부 운영체제는 태스크라고도 부른다.) 수많은 프로세스가 동시에(= 매우 빠르게 번갈아) 생성·실행·삭제되며, 각 프로세스의 상황도 다양하다(입출력장치 필요, CPU 급히 필요, 당장 실행 불가 등). 운영체제는 이 다양한 프로세스를 일목요연하게 관리한다.

> 이와 관련해 앞으로 프로세스와 스레드, 프로세스 동기화, 교착 상태(데드락)와 그 해결을 배운다.

#### 2. 자원 접근 및 할당

| 자원 | 관리 방법 | 학습 내용 |
| --- | --- | --- |
| **CPU** | CPU 스케줄링 | 어떤 프로세스를 먼저·얼마나 오래 실행할지 공평하게 할당 |
| **메모리** | 페이징·스와핑 | 빈 공간을 어떻게 찾을지, 메모리보다 큰 프로세스를 어떻게 실행할지 |
| **입출력장치** | 인터럽트 서비스 루틴(ISR) | 응용 프로그램이 입출력장치에 접근·할당받도록 함 (ISR도 커널 영역 코드) |

### 가상 머신

> **참고** — 이 항목은 강의 범위 밖의 보충 내용이다. **가상 머신(Virtual Machine)**은 실제 하드웨어를 소프트웨어적으로 흉내 내어, 하나의 물리적 컴퓨터 위에서 여러 개의 가상 컴퓨터(각자의 운영체제)를 실행할 수 있게 하는 기술이다. 물리 자원을 가상화해 나누어 쓰며, 하이퍼바이저가 이를 관리한다. 자세한 내용은 별도 자료를 참고할 것.
