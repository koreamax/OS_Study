# 스레드 & CPU 스케줄링 개요

## 10-3 스레드

(여기서 다루는 스레드는 CPU 하드웨어 스레드가 아니라 **소프트웨어적인 스레드**다.)

### 프로세스 vs 스레드

**스레드(thread)**는 **프로세스를 구성하는 실행 흐름의 단위**다. 하나의 프로세스는 하나 이상의 스레드를 가질 수 있다.

- **단일 스레드 프로세스** — 실행 흐름이 하나뿐인 프로세스 (지금까지 상정했던 프로세스)
- **멀티스레드 프로세스** — 실행 흐름이 여러 개인 프로세스. 하나의 프로세스를 스레드 개수만큼 병행 실행할 수 있다. (예: 웹브라우저에서 화면 출력·입력·검색을 각각 별도 스레드로 처리)

#### 스레드의 구성 요소

각 스레드는 프로세스의 각기 다른 부분을 실행하기 위해, **실행에 필요한 최소한의 정보**를 각자 유지한 채 실행된다.

- **스레드 ID**
- **프로그램 카운터를 비롯한 레지스터 값**
- **스택**

> **핵심** — 스레드들은 실행에 필요한 최소한의 정보(스레드 ID·레지스터·스택)만 각자 갖고, 나머지 **프로세스의 자원(코드·데이터·힙 영역, 열린 파일 등)은 서로 공유**하면서 실행된다. 스레드마다 별도의 코드 영역·데이터 영역이 있는 것이 아니다.

> **참고 — 리눅스의 관점** — 리눅스는 프로세스와 스레드를 임의로 구분하지 않고 **태스크(task)**라는 용어로 통일해 다룬다. (리누스 토르발스는 둘을 "실행 문맥이라는 점에서 같은 것"으로 본다.) 또한 최근 운영체제는 CPU에 작업을 전달할 때 프로세스 통째가 아니라 **스레드 단위**로 할당하는 경우가 많다. (다만 전통적 전공서는 여전히 프로세스 단위로 기술한다.)

### 멀티프로세스 vs 멀티스레드

같은 작업을 하는 단일 프로세스 여러 개를 실행하는 것(**멀티프로세스**)과, 하나의 프로세스를 여러 스레드로 실행하는 것(**멀티스레드**)의 차이를 살펴본다.

> 예) "Hello OS"를 출력하는 프로그램을, ① fork로 세 번 실행하면 동일한 프로세스 3개가 각각 출력 → 화면에 3번 출력. ② 한 프로세스를 스레드 3개로 실행해도 → 화면에 3번 출력. **결과는 같지만 내부 구조가 다르다.**

**핵심 차이 — 자원 공유 여부**

| 구분 | 멀티프로세스 | 멀티스레드 |
| --- | --- | --- |
| 자원 공유 | 기본적으로 **공유하지 않음**(남남처럼 독립 실행) | 같은 프로세스 내 **자원을 공유** |
| PCB | 프로세스마다 별도(PCB 여러 개) | 프로세스 하나이므로 PCB 하나 |
| 협력·통신 | 상대적으로 불리 | 자원 공유로 유리 |
| 한 실행 흐름에 문제 발생 시 | 다른 프로세스에 영향 적음 | **전체 프로세스에 영향**을 줄 수 있음 |

- fork로 프로세스를 복제하면 코드·데이터·힙 영역 등 모든 자원이 자식 프로세스로 복제되어, (메모리 주소를 제외하면) 동일한 프로세스가 통째로 메모리에 적재된다.
- 반면 스레드들은 실행에 꼭 필요한 값만 각자 갖고 프로세스의 자원을 공유한다.

> **쓰기 시 복사(Copy-on-Write)** — fork 직후 실제로는 같은 프로세스를 통째로 중복 저장하지 않으면서도 자원을 공유하지 않게 하는 기법이 있다. (페이징에서 다룸.)

**프로세스 간 통신 (IPC)**

프로세스끼리는 기본적으로 자원을 공유하지 않지만, 통신은 가능하다. 이를 **IPC(Inter-Process Communication)**라 한다. (스레드만큼 쉽지 않을 뿐이다.) 같은 컴퓨터 내 프로세스·스레드끼리 데이터를 주고받는 것도 통신에 해당한다.

- **파일을 통한 통신** — 한 프로세스가 파일에 쓰고, 다른 프로세스가 그 파일을 읽음
- **공유 메모리를 통한 통신** — 프로세스들이 공유하는 메모리 영역을 두고 데이터를 주고받음
- 이외 소켓, 파이프 등 다양한 기법이 있다.

## 11-1 CPU 스케줄링 개요

**CPU 스케줄링**은 운영체제가 프로세스들에게 공정하고 합리적으로 CPU 자원을 배분하는 것이다. 컴퓨터 전체 성능과 직결되는 중요한 문제로, CPU를 제대로 분배하지 못하면 급한 프로세스가 실행되지 못하는 상황이 생길 수 있다.

### 프로세스 우선순위

CPU를 원하는 순서대로 무작정 나눠주는 것은 좋은 방법이 아니다. **프로세스마다 우선순위가 다르기** 때문이다. 빨리 처리해야 하는 프로세스가 있고 천천히 처리해도 되는 프로세스가 있다.

대부분의 프로세스는 CPU와 입출력장치를 번갈아 사용하며 실행되는데, 어느 쪽을 많이 쓰느냐에 따라 나뉜다.

| 종류 | 특징 | 우선순위 |
| --- | --- | --- |
| **입출력 집중 프로세스(I/O bound)** | 입출력 작업이 많음 (예: 비디오 재생, 디스크 백업). 대기 상태에 더 오래 머무름 | **높음** |
| **CPU 집중 프로세스(CPU bound)** | CPU 작업이 많음 (예: 컴파일, 복잡한 수학 연산, 그래픽 처리). 실행 상태에 더 오래 머무름 | 낮음 |

> **입출력 집중 프로세스의 우선순위가 높은 이유** — 입출력 집중 프로세스는 잠깐 CPU를 쓴 뒤 곧 대기 상태로 들어가 한동안 CPU를 쓰지 않는다. 따라서 이런 프로세스의 우선순위를 높여 빨리 처리하고, 그 사이 CPU 집중 프로세스에 CPU를 몰아주는 것이 효율적이다.

우선순위는 프로세스의 **PCB에 저장**되며, 운영체제는 이를 보고 어떤 프로세스가 CPU를 얼마나 사용할지 결정한다. 우선순위가 높은 프로세스는 더 빨리·더 자주 실행된다. (리눅스의 `nice` 값, 윈도우의 프로세스 익스플로러 등으로 우선순위를 확인·설정할 수 있다.)

### 스케줄링 큐

운영체제가 다음에 CPU를 사용할 프로세스를 정하려고 매번 모든 프로세스의 PCB를 뒤지는 것은 비효율적이다. 그래서 **스케줄링 큐(scheduling queue)**를 사용한다.

- 스케줄링 큐는 **특정 자원을 이용하고 싶은 프로세스들이 서는 줄**이다. 운영체제는 자원을 원하는 프로세스를 해당 큐에 삽입해 줄을 세운다.
- 자원마다 큐가 있어(CPU 큐, 하드디스크 큐, 프린터 큐 등) 여러 종류가 존재한다.

> **참고** — 자료구조로서의 큐(queue)는 원래 먼저 삽입된 데이터가 먼저 나가는 **선입선출(FIFO)** 구조지만, 스케줄링 큐는 반드시 선입선출을 따르지 않는다. (우선순위가 반영되기 때문)

**대표적인 두 스케줄링 큐**

| 큐 | 대기하는 프로세스 |
| --- | --- |
| **준비 큐(ready queue)** | CPU를 이용하고자 하는(준비 상태의) 프로세스들의 줄 |
| **대기 큐(waiting queue)** | 입출력장치를 이용하고자 하는(대기 상태의) 프로세스들의 줄 |

> 먼저 큐에 삽입되었다고 반드시 먼저 처리되는 것은 아니며, **우선순위가 높은 프로세스가 먼저** CPU를 이용한다.

**전체 흐름 (프로세스 상태 + 큐)**

1. 여러 프로세스가 **준비 큐**에서 CPU 할당을 기다린다.
2. 자기 차례가 되면 **디스패치**되어 CPU를 할당받아 **실행 상태**가 된다.
3. 할당된 시간을 다 쓰면(**타이머 인터럽트**) 다시 **준비 큐**로 돌아간다.
4. 실행 중 입출력 작업을 만나면 **대기 큐**에 들어가 **대기 상태**가 된다.
5. 입출력이 완료되면(**입출력 완료 인터럽트**) 대기 큐에서 나와 다시 **준비 큐**로 들어간다.

> 대기 큐는 보통 **입출력장치별로** 있어, 같은 장치를 요구한 프로세스들은 같은 큐에서 대기한다. 입출력이 완료되면 운영체제는 대기 큐에서 해당 PCB를 찾아 상태를 대기→준비로 바꾸고, 대기 큐에서 준비 큐로 옮긴다.

### 선점형 vs 비선점형 스케줄링

어떤 프로세스가 CPU를 사용하는 도중, 더 급한 프로세스가 CPU를 요청했을 때 어떻게 처리하느냐에 따라 두 방식으로 나뉜다.

| 구분 | 선점형 스케줄링 (preemptive) | 비선점형 스케줄링 (non-preemptive) |
| --- | --- | --- |
| 방식 | 실행 중인 프로세스로부터 운영체제가 CPU 자원을 **뺏어** 다른 프로세스에 넘길 수 있음 | 한 프로세스가 자원을 점유하면, **종료되거나 대기 상태가 되기 전까지** 다른 프로세스가 끼어들 수 없음 |
| 장점 | 한 프로세스의 자원 독점을 막고 **골고루 배분** | 문맥 교환 **오버헤드가 적음** |
| 단점 | 문맥 교환이 잦아 **오버헤드 발생** | 모든 프로세스가 골고루 자원을 이용하기 **어려움** |

> 지금까지 설명한, 프로세스마다 정해진 시간만큼 CPU를 쓰다가 **타이머 인터럽트**가 발생하면 운영체제가 CPU를 뺏어 다음 프로세스에 넘기는 방식이 바로 **선점형 스케줄링**의 일종이다. 두 방식의 장단점은 서로 뒤집은 관계다.
