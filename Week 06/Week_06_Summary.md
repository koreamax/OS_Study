# Chapter 05-2 ~ 05-3. 명령어 병렬 처리 기법과 CISC·RISC

## 05-2 명령어 병렬 처리 기법

CPU를 빠르게 설계하는 것뿐 아니라, CPU가 **시간 낭비 없이** 명령어를 빠르게 실행하도록 하는 것도 중요하다. 이를 위한 대표적인 기법이 명령어 병렬 처리 기법이다.

### 명령어 파이프라인

하나의 명령어가 처리되는 과정을 비슷한 시간 간격의 단계로 나눌 수 있다. (전공서에 따라 2~5단계로 나누며, 절대적이지 않다.) 여기서는 4단계로 본다.

1. 명령어 인출(fetch) — 메모리에서 명령어를 가져옴
2. 명령어 해석(decode)
3. 명령어 실행(execute)
4. 결과 저장(store) — 필요시 결과를 메모리에 저장

**같은 단계가 겹치지만 않으면** CPU는 각 단계를 동시에 실행할 수 있다. 즉 여러 명령어를 겹쳐서 병렬로 처리할 수 있는데, 이를 **명령어 파이프라인**이라고 한다. (컨베이어 벨트처럼 인출→해석→실행→저장을 멈추지 않고 반복하는 것에 비유할 수 있다.)

> 예) 한 시점에 명령어 1은 실행, 명령어 2는 해석, 명령어 3은 인출 — 세 명령어가 겹쳐 실행된다. 파이프라인을 쓰지 않으면(하나를 끝까지 처리한 뒤 다음 처리) 전체 시간이 훨씬 길어진다.

#### 파이프라인 위험(pipeline hazard)

파이프라인이 항상 이상적으로 동작하지는 않는다. 명령어를 동시에 처리할 수 없어 성능 향상에 실패하는 경우를 **파이프라인 위험**이라 하며, 세 종류가 있다.

| 종류 | 원인 |
| --- | --- |
| **데이터 위험(data hazard)** | 명령어 간의 **의존성** 때문. 이전 명령어 실행이 끝나야만 다음 명령어를 실행할 수 있는 경우 |
| **제어 위험(control hazard)** | **프로그램 카운터의 갑작스러운 변화**(점프, 콜, 인터럽트 등) 때문 |
| **구조 위험(structural hazard)** | 서로 다른 명령어가 **같은 CPU 부품**(ALU, 레지스터 등)을 사용하려 할 때 |

**데이터 위험 예시**

```
명령어 1: R1 ← R2 + R3   (R2와 R3를 더해 R1에 저장)
명령어 2: R4 ← R1 + R5   (R1과 R5를 더해 R4에 저장)
```

명령어 2는 R1 값이 필요하므로, 명령어 1이 실행되어 R1이 결정된 뒤에야 실행할 수 있다. 따라서 둘을 마냥 겹쳐 실행할 수 없다.

**제어 위험 예시** — 순차적으로 10→11→12번지를 겹쳐 실행 중인데 `JUMP 60` 같은 명령어로 60번지로 분기하면, 이미 인출·해석하던 11·12번지 명령어는 헛수고가 된다. (이를 방지하려고 분기할 주소를 미리 예측하는 **분기 예측(branch prediction)** 기술도 있다.)

### 슈퍼스칼라

**여러 개의 명령어 파이프라인을 두는 구조**다. 한 번에 여러 명령어를 인출·해석·실행·저장할 수 있어 더 많이 겹쳐 실행할 수 있다. (멀티스레드 프로세서처럼 한 번에 여러 명령어를 가져올 수 있는 CPU에서 활용 가능.)

> 이론적으로는 파이프라인 개수에 비례해 처리 속도가 증가하지만, 현실적으로는 그렇지 않다. 파이프라인이 늘어나면 관리할 파이프라인이 많아지고 **파이프라인 위험도 함께 증가**하기 때문이다.

### 비순차적 명령어 처리(OoOE)

**Out-of-Order Execution**의 약자로, 파이프라인 중단을 막기 위해 명령어를 **순차적으로 처리하지 않는** 기법이다. **의존성이 없는 명령어들의 순서를 바꿔** 파이프라인이 원활히 동작하도록 한다.

> 예) 다음에서 3번은 1·2번 결과(100·101번지 값)에 의존하므로 1·2번이 끝나야 실행 가능하다. 이를 순차 처리하면 파이프라인이 중단된다.
> ```
> 1: M[100] ← 1
> 2: M[101] ← 2
> 3: M[102] ← M[100] + M[101]   ← 1, 2에 의존
> 4: ...
> ```
> 3번을 의존성 없는 다른 명령어들 뒤로 미루면, 3번을 인출하는 시점엔 이미 100·101번지 값이 결정되어 있어 중단을 방지할 수 있다.

단, 아무 명령어나 순서를 바꿀 수는 없다. **순서를 바꿔도 전체 실행 결과에 영향이 없는(의존성이 없는) 명령어**만 재배치할 수 있다. 비순차적 처리를 지원하는 CPU는 어떤 명령어를 바꿔도 되는지 스스로 판단한다.

## 05-3 CISC와 RISC

### 명령어 집합

세상의 모든 CPU가 같은 명령어를 실행하지는 않는다. CPU 제조사·종류마다 이해하는 명령어의 종류, 연산 방식, 주소 지정 방식이 다를 수 있다.

- **명령어 집합(instruction set)** / **명령어 집합 구조(ISA, Instruction Set Architecture)**: CPU가 이해할 수 있는 명령어들의 모음.
- CPU는 자신의 명령어 집합에 속한 명령어만 이해할 수 있다. (그래서 인텔 PC에서 만든 실행 파일을 그대로 아이폰·맥북에 옮겨 실행할 수 없다. 같은 소스 코드도 CPU 종류에 따라 컴파일 결과인 어셈블리어가 달라진다.)

> ISA가 다르면 명령어 해석 방식, 레지스터의 종류·개수, 파이프라이닝 난이도 등 많은 것이 함께 달라진다. 그래서 ISA를 **하드웨어와 소프트웨어 사이의 약속**으로 보기도 한다.

현대 명령어 집합 구조의 가장 대표적인 두 종류가 **CISC**와 **RISC**다.

### CISC (Complex Instruction Set Computer)

**복잡한 명령어 집합을 활용하는 컴퓨터(CPU).** 인텔·AMD의 `x86`, `x86-64`가 대표적이다.

- 복잡하고 다양한 명령어를 사용하며, 명령어의 크기와 형태가 다양하다 → **가변 길이 명령어**.
- 명령어 하나하나가 강력한 기능을 제공하고 종류도 다양해, **상대적으로 적은 수의 명령어**로 프로그램을 실행할 수 있다.
- 메모리를 아껴야 했던 시절에 인기가 높았다.

> **단점 — 명령어 파이프라이닝에 불리하다.** 명령어가 복잡해 크기와 실행 시간이 일정하지 않고, 하나의 복잡한 명령어를 실행하는 데 여러 클럭 주기가 필요한 경우가 많다. 파이프라이닝은 명령어 실행 시간이 정형화되어 있어야 잘 동작하므로 CISC에는 불리하다. 게다가 복잡한 명령어 대부분은 사용 빈도가 낮다. (CISC 명령어 중 약 20%가 전체 사용량의 약 80%를 차지한다는 연구도 있다.)

### RISC (Reduced Instruction Set Computer)

**명령어 집합의 수를 줄인 컴퓨터(CPU).** ARM이 대표적이다.

- 명령어 종류가 적고, 명령어가 짧으며 가급적 **1클럭 내로 실행되는 규격화된 명령어**(고정 길이 명령어)를 사용한다.
- 규격화된 명령어 덕분에 **명령어 파이프라이닝에 유리하다.**
- **메모리 접근을 최소화**하는 경향이 있다. 메모리 접근을 LOAD(읽기)·STORE(쓰기) 두 명령어로 제한할 만큼 줄이고, 대신 **레지스터를 적극 활용**한다. → 범용 레지스터 종류가 CISC보다 많은 편.
- 명령어 종류가 적어, 같은 프로그램이라도 **더 많은 명령어**로 동작한다. (CISC 한 줄로 할 일을 RISC는 여러 줄로 수행.)

### CISC와 RISC 정리

| 구분 | CISC | RISC |
| --- | --- | --- |
| 명령어 | 복잡하고 다양함 | 단순하고 적은 종류 |
| 명령어 길이 | 가변 길이 | 고정 길이 |
| 실행 주기 | 일정하지 않음(여러 클럭) | 정형화됨(가급적 1클럭) |
| 주소 지정 방식 | 다양함 | 적음 (메모리 접근 최소화) |
| 프로그램의 명령어 수 | 적음 | 많음 |
| 파이프라이닝 | 불리 | 유리 |
| 대표 예 | x86, x86-64 (인텔·AMD) | ARM |

> **참고 — 현대 CPU의 실제 설계** — 위 내용은 이론적 구분이다. 실제로는 CISC 기반 CPU도 파이프라이닝이 필수이므로, 내부적으로 복잡한 명령어를 **마이크로 명령어**라는 더 작은 단위로 잘게 쪼개 가급적 1클럭 내에 실행한다. 즉 현대 CISC 프로세서도 내부적으로는 RISC처럼 동작하는 경우가 많다.
