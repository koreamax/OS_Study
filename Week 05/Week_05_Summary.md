# Chapter 04-3 ~ 05-1. 명령어 사이클·인터럽트와 CPU 성능 향상 기법

## 04-3 명령어 사이클과 인터럽트

CPU는 메모리에 있는 프로그램을 **일정한 흐름(주기)**에 따라 처리한다. 이 정해진 흐름을 **명령어 사이클**이라 하고, 이 흐름을 끊는 신호를 **인터럽트**라 한다.

### 명령어 사이클

CPU는 프로그램 속 명령어들을 일정한 주기를 반복하며 실행한다. 이 주기가 명령어 사이클이다.

| 사이클 | 설명 |
| --- | --- |
| **인출(fetch) 사이클** | 메모리에 저장된 값을 CPU 내부로 가져오는 주기 |
| **실행(execution) 사이클** | 가져온 명령어를 실행하는 주기 |
| **간접(indirect) 사이클** | 인출 후에도 바로 실행할 수 없어 메모리에 추가로 접근해야 할 때 거치는 주기 |

일반적으로 CPU는 **인출 → 실행 → 인출 → 실행 …**을 반복하며 동작한다. 단, 간접 주소 지정 방식처럼 인출만으로는 실행이 불가능하고 추가 메모리 접근이 필요한 경우에는 인출 후 **간접 사이클**에 들어갔다가 실행 사이클로 넘어간다.

### 인터럽트

**인터럽트(interrupt)**는 CPU의 정해진 실행 흐름을 방해(중단)하는 신호다. CPU가 얼른 처리해야 할 작업이 생겼을 때 발생한다. (회사에서 "하던 일 잠깐 멈추고 이 급한 것부터 처리해 줘"라고 하는 상황과 같다.)

인터럽트는 크게 두 종류로 나뉜다.

| 종류 | 다른 이름 | 발생 시점 |
| --- | --- | --- |
| **동기 인터럽트** | 예외(exception) | CPU가 예기치 못한 상황을 접했을 때 |
| **비동기 인터럽트** | 하드웨어 인터럽트 | 주로 입출력장치 등 하드웨어가 보낼 때 |

#### 동기 인터럽트(예외)

CPU가 명령어를 실행하다 **예기치 못한 상황**을 만났을 때 발생한다. 예: 접근한 메모리 주소에 원하는 데이터가 없음, 0으로 나누기, 실행할 수 없는 명령어, 디버깅 등.

#### 비동기 인터럽트(하드웨어 인터럽트)

주로 입출력장치 등 다른 하드웨어가 보내는 인터럽트로, **알림(notification)** 역할에 가깝다. (세탁기·전자레인지의 완료 알림음을 떠올리면 된다. 키보드 입력, 마우스 클릭도 하드웨어 인터럽트의 일종이다.)

> **하드웨어 인터럽트를 쓰는 이유** — CPU가 입출력 작업 도중에도 효율적으로 명령어를 처리하기 위해서다. 입출력장치의 작업은 CPU보다 느리다. 인터럽트가 없다면 CPU는 작업 완료 여부를 주기적으로 직접 확인해야 한다("다 됐나? 아니네"의 반복). 인터럽트가 있으면 CPU는 다른 일을 하다가 완료 알림을 받았을 때만 처리하면 된다.

### 하드웨어 인터럽트의 처리 순서

(인터럽트 종류와 무관하게 대부분의 인터럽트 처리 순서는 이 틀에서 크게 벗어나지 않는다.)

1. 입출력장치가 CPU에 **인터럽트 요청 신호**를 보낸다.
2. CPU는 **실행 사이클이 끝나고 명령어를 인출하기 전에** 항상 인터럽트 여부를 확인한다.
3. 인터럽트 요청을 확인하면 **인터럽트 플래그**를 통해 현재 인터럽트를 받아들일 수 있는지 확인한다.
4. 받아들일 수 있다면 CPU는 지금까지의 작업을 **백업**한다.
5. **인터럽트 벡터**를 참조해 **인터럽트 서비스 루틴(ISR)**을 실행한다.
6. ISR 실행이 끝나면 백업해 둔 작업을 **복구**해 원래 작업을 재개한다.

#### 핵심 용어

| 용어 | 의미 |
| --- | --- |
| **인터럽트 요청 신호** | 인터럽트를 보내는 주체가 "끼어들어도 되는지" CPU에 보내는 요청 신호 |
| **인터럽트 플래그** | 플래그 레지스터 안의 값. 현재 인터럽트를 받아들일 수 있는지 여부를 나타냄 |
| **인터럽트 서비스 루틴(ISR)** | 해당 인터럽트를 어떻게 처리할지가 적힌 프로그램(메모리에 저장). 인터럽트를 보내는 주체마다 다름 |
| **인터럽트 벡터** | 각 인터럽트를 식별하는 정보. ISR의 시작 주소를 알기 위해 사용. (벡터들을 표로 모은 것이 **인터럽트 벡터 테이블**) |

> **막을 수 없는 인터럽트(NMI)** — 모든 인터럽트를 인터럽트 플래그로 막을 수 있는 것은 아니다. 하드웨어 고장, 정전처럼 반드시 처리해야 하는 긴급한 인터럽트는 막을 수 없으며, 이를 **넌 마스커블 인터럽트(NMI, Non-Maskable Interrupt)**라고 한다. 대부분의 입출력장치 하드웨어 인터럽트는 인터럽트 플래그로 막을 수 있다.

#### 백업과 복구 (스택 활용)

ISR도 프로그램이므로 실행하면서 프로그램 카운터 등 레지스터를 사용한다. 따라서 인터럽트로 점프하기 전, **현재 레지스터 값들을 스택에 백업**해 둔다. ISR 실행이 끝나면 스택에서 값을 복구해 다음에 실행할 명령어부터 작업을 이어간다.

> **인터럽트가 포함된 명령어 사이클** — 실행 사이클이 끝난 뒤 인터럽트 여부를 확인하고, 인터럽트가 발생했다면 **인터럽트 사이클**이 추가된다.

### 예외의 종류

동기 인터럽트(예외)는 다른 전공서의 표현을 빌리면 다음 네 가지로 나뉜다. (자세한 내용은 운영체제에서 다룬다.)

| 종류 | 설명 |
| --- | --- |
| **폴트(fault)** | 예외를 처리한 직후, 예외가 발생한 명령어부터 다시 실행 |
| **트랩(trap)** | 예외를 처리한 직후, 예외가 발생한 명령어의 다음 명령어부터 실행 (주로 디버깅) |
| **중단(abort)** | CPU가 실행을 더 이어갈 수 없는 심각한 오류 발견 시 |
| **소프트웨어 인터럽트** | 시스템 호출 등이 발생했을 때 |

# Chapter 05. CPU 성능 향상 기법

## 05-1 빠른 CPU를 위한 설계 기법

더 빠른 CPU를 만드는 대표적인 방법으로 **클럭 높이기**, **코어 수 늘리기(멀티코어)**, **스레드 수 늘리기(멀티스레드)**가 있다.

### 클럭

컴퓨터 부품은 **클럭(clock) 신호**에 맞춰 일사불란하게 움직이고, CPU는 명령어 사이클에 따라 명령어를 실행한다. 따라서 클럭 신호를 빠르게 반복하면 명령어 사이클도 그만큼 빠르게 반복된다.

- 클럭 속도는 CPU의 속도 단위로 간주되기도 하며, **헤르츠(Hz)** 단위로 측정한다. (1Hz = 클럭이 1초에 1번 반복)
- 예) `2.5GHz`는 클럭이 1초에 약 25억 번, `4.9GHz`는 약 49억 번 반복됨을 뜻한다.
- 클럭 속도는 항상 일정하지 않다. CPU는 필요에 따라 클럭을 빠르게/느리게 조절한다.

> **클럭을 무작정 높일 수 없는 이유** — 필요 이상으로 클럭을 높이면 **발열**이 심해진다. (게임·영상 편집 등 고부하 작업 시 CPU가 뜨거워지는 현상.)

### 코어와 멀티코어

현대적 관점에서 CPU 용어를 재해석해야 한다.

- 전통적으로 명령어를 인출·해석·실행하는 부품은 컴퓨터 내부에 **하나**만 있었고, 이것이 곧 CPU였다.
- 오늘날에는 이 부품이 CPU 안에 **여러 개** 존재할 수 있게 되었고, 그 각각을 **코어(core)**라고 부른다.

| 용어 | 의미 |
| --- | --- |
| **코어** | CPU(프로세서) 내에서 명령어를 인출·해석·실행하는 부품 |
| **멀티코어** | 두 개 이상의 코어를 가진 CPU (싱글코어, 듀얼코어, 트리플코어, 쿼드코어, 헥사코어 …) |

> **코어 수에 비례해 빨라지지 않는다** — 조별 과제처럼, 작업이 각 코어에 적절히 분배되지 않으면 코어 수만큼 속도가 오르지 않는다. 또 처리할 작업량이 코어 수에 비해 지나치게 적으면(요리사는 많은데 도시락은 적은 경우) 성능 향상 효과가 작다. 따라서 **명령어를 각 코어에 얼마나 적절히 분배하느냐**가 중요하다.

### 스레드와 멀티스레드

**스레드(thread)**는 혼동을 줄이기 위해 하드웨어적 스레드와 소프트웨어적 스레드로 나누어 이해하는 것이 좋다.

#### 하드웨어 스레드

**하나의 코어가 동시에 처리하는 명령어의 단위**다.

- 하나의 코어가 한 번에 하나의 명령어를 처리 → 1코어 1스레드
- 코어 2개 × 각 코어가 동시에 2개 명령어 처리 → **2코어 4스레드**
- 하나의 코어가 여러 명령어를 동시에 처리할 수 있는 CPU를 **멀티스레드 프로세서**라 한다. (인텔의 멀티스레드 기술을 **하이퍼스레딩**이라 부른다.)

> 예) `8코어 16스레드`는 명령어 실행 부품(코어)이 8개이고, 각 코어가 2개씩 명령어를 동시에 처리해 총 16개 명령어를 동시에 처리할 수 있음을 의미한다.

**논리 프로세서** — 메모리 속 프로그램 입장에서는 실제 코어 수를 알 수 없고, 동시에 몇 개의 명령어가 처리되는지(= 하드웨어 스레드 수)만 느낀다. 그래서 하드웨어 스레드를 **논리 프로세서**라고도 부른다. (윈도우 작업 관리자의 "논리 프로세서" 수가 이것이다.)

**멀티스레드 프로세서 설계의 핵심은 레지스터다.** 하나의 명령어를 실행하는 데 필요한 레지스터 묶음(프로그램 카운터, 명령어 레지스터 등 = 레지스터 세트)을 하나의 코어가 여러 개 가지면, 그 코어는 여러 명령어를 동시에 처리할 수 있다. (예: 프로그램 카운터가 2개면 다음 실행할 명령어 주소를 2개 지정 가능, 스택 포인터가 2개면 스택 2개를 관리 가능.)

#### 소프트웨어 스레드

**하나의 프로그램에서 독립적으로 실행되는 단위**다.

- 단일 스레드: 한 프로그램이 하나의 실행 흐름으로 순차 실행
- 멀티스레드: 하나의 프로그램에서 여러 실행 흐름이 동시에 실행
- 예) 워드프로세서에서 ① 입력 내용 화면 표시 ② 맞춤법 검사 ③ 자동 임시 저장을 각각 소프트웨어 스레드로 만들어 동시에 실행

> **중요한 맹점** — 하드웨어 스레드가 하나(1코어 1스레드)여도 소프트웨어 스레드는 여러 개 만들 수 있다. CPU가 여러 프로그램·스레드를 엄청나게 빠르게 번갈아 실행해, 사용자 눈에는 동시에 실행되는 것처럼 보이기 때문이다. (소프트웨어 스레드는 운영체제 파트에서 멀티프로세스와 함께 자세히 다룬다.)

| 구분 | 하드웨어 스레드 | 소프트웨어 스레드 |
| --- | --- | --- |
| 정의 | 하나의 코어가 동시에 처리하는 명령어 단위 | 하나의 프로그램에서 독립적으로 실행되는 단위 |
| 관점 | CPU 하드웨어 | 프로그램(소프트웨어) |
| 비고 | 논리 프로세서라고도 함 | 하드웨어 스레드가 1개여도 여러 개 생성 가능 |
