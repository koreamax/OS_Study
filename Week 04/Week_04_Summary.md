# Chapter 04. CPU의 작동 원리

CPU 내부에는 여러 부품이 있지만 크게 **ALU**, **제어장치**, 여러 개의 **레지스터**로 이루어진다. 이 장에서는 각 부품이 어떤 정보를 주고받는지 살펴본다. (각 부품을 구성하는 회로 자체는 다루지 않는다.)

## 04-1 ALU와 제어장치

### ALU

ALU는 **계산을 담당하는 장치(계산기)**다. 계산에 필요한 정보를 받아들여 연산을 수행하고 결과를 내보낸다.

**받아들이는 정보**

| 정보 | 출처 |
| --- | --- |
| **피연산자** | 레지스터 |
| **제어 신호(수행할 연산)** | 제어장치 |

> 예) `1 + 2`를 계산하려면 피연산자(1, 2)와 수행할 연산(더하기)이 필요하다. 피연산자는 레지스터로부터, 수행할 연산은 제어장치의 제어 신호로 받는다.

**내보내는 정보**

1. **결과값** — 연산 결과(숫자, 문자, 주소 등 0과 1로 표현된 정보)를 **레지스터**에 저장한다.
   - 메모리가 아니라 레지스터에 담는 이유: CPU가 레지스터에 접근하는 속도가 메모리에 접근하는 속도보다 빠르기 때문이다. 임시 결과를 레지스터에 담은 뒤 메모리에 쓰거나 다른 연산에 사용한다.
2. **플래그** — 연산 결과에 대한 **부가 정보**. (이 장에서 매우 중요한 개념이다.)

#### 플래그

**플래그**는 연산 결과에 대한 부가 정보이며, **플래그 레지스터**라는 특수한 레지스터에 저장된다.

> 2장에서 이진수로 음수를 표현할 때 양수와 음수의 0·1 표현이 같아 보이는 문제를 다뤘다. 이를 구분해 주는 것이 바로 플래그다. CPU 내부의 모든 숫자가 양수/음수 깃발(플래그)을 들고 다닌다고 이해하면 된다.

대부분의 CPU가 공통으로 포함하는 플래그 종류는 다음과 같다.

| 플래그 | 의미 |
| --- | --- |
| **부호 플래그** | 연산 결과가 양수인지 음수인지 |
| **제로 플래그** | 연산 결과가 0인지 아닌지 |
| **캐리 플래그** | 연산 결과에 올림수나 빌림수가 발생했는지 (예: `9+1`의 올림수 1) |
| **오버플로우 플래그** | 연산 결과가 레지스터 크기보다 커졌는지 |
| **인터럽트 플래그** | 인터럽트가 가능한지 여부 (이후 학습) |
| **슈퍼바이저 플래그** | 커널 모드/사용자 모드 실행 여부 (운영체제에서 학습) |

> 예) 부호 플래그가 1이면 CPU는 해당 연산 결과를 음수로 인식하고, 제로 플래그가 1이면 결과를 0으로 인식한다.

**ALU 내부 회로** — ALU 안에는 덧셈을 위한 **가산기**, 뺄셈을 위한 **보수기**, 비트를 이동시키는 **시프터**, **오버플로우 검출기** 등 다양한 계산 회로가 있다. (회로 자체는 이 장에서 다루지 않는다.)

### 제어장치

제어장치는 **제어 신호를 발생시키고 명령어를 해석하는 장치**다.

**받아들이는 정보**

| 정보 | 설명 |
| --- | --- |
| **클럭(clock) 신호** | 컴퓨터의 모든 부품을 일정한 박자에 맞춰 일사불란하게 움직이게 하는 시간 단위 |
| **해석할 명령어** | 명령어 레지스터에 저장된, 해석 대상 명령어 |
| **플래그 값** | 플래그 레지스터에 담긴 연산 결과 부가 정보 (명령어 해석에 필요) |
| **(외부) 제어 신호** | 입출력장치 등 주변 장치가 발생시켜 CPU로 전달된 제어 신호 |

> **클럭** — '시계(clock)'가 똑딱똑딱 움직이듯, 클럭 신호는 일정한 박자로 발생한다. 명령어들은 이 클럭 신호에 맞춰 수행된다. 클럭 신호를 발생시키는 부품은 따로 있다.

**내보내는 정보** — 제어장치는 **제어 신호**를 내보낸다. 제어 신호는 전달 대상에 따라 나뉜다.

| 구분 | 대상 | 내보내는 제어 신호 |
| --- | --- | --- |
| **CPU 내부** | 레지스터 | 레지스터 간 정보 이동·저장 등 레지스터를 움직이는 지시 |
| | ALU | 수행할 연산 지시 |
| **CPU 외부** | 메모리 | 메모리 읽기/쓰기 지시 |
| | 입출력장치 | 입출력장치 읽기/쓰기/테스트 지시 |

> 제어 신호를 발생시키는 부품이 CPU만은 아니다. 입출력장치 등 주변 장치도 제어 신호를 발생시킬 수 있다.

## 04-2 레지스터

레지스터는 CPU 내부의 작은 임시 저장 장치다. 프로그램의 명령어와 데이터 관련 값이 실행 전후로 레지스터에 저장되므로, 레지스터 값을 관찰하면 프로그램의 실행 순서와 원리를 가장 낮은 단계에서 파악할 수 있다. (특히 시스템·임베디드·보안 분야 개발자에게 중요하다.)

레지스터의 종류와 개수, 이름은 CPU마다 다르지만, 대부분의 전공서와 CPU가 공통으로 포함하는 레지스터가 있다.

### 반드시 알아야 할 레지스터

| 레지스터 | 저장하는 값 |
| --- | --- |
| **프로그램 카운터(PC)** | 메모리에서 가져올(다음에 실행할) 명령어의 주소. (인스트럭션 포인터라고도 함) |
| **명령어 레지스터(IR)** | 해석할 명령어. 메모리에서 막 읽어 들인 명령어를 저장 |
| **메모리 주소 레지스터(MAR)** | 메모리의 주소. 주소 버스로 주소를 내보낼 때 거침 |
| **메모리 버퍼 레지스터(MBR)** | 메모리와 주고받을 값(명령어·데이터). 데이터 버스로 주고받을 때 거침 |
| **플래그 레지스터** | 연산 결과나 CPU 상태에 대한 부가 정보(플래그) |
| **범용 레지스터** | 주소·명령어·데이터 등 무엇이든 자유롭게 저장. 보통 여러 개 존재 |
| **스택 포인터** | 스택의 꼭대기 주소 (스택 주소 지정 방식에 사용) |
| **베이스 레지스터** | 기준 주소 (변위 주소 지정 방식에 사용) |

#### 명령어 실행 흐름 예시

메모리 1000번지부터 프로그램이 저장되어 있고, 첫 명령어부터 실행한다고 하자.

1. **PC**에 `1000`(다음에 실행할 명령어 주소)이 저장된다.
2. PC의 값 `1000`이 **MAR**로 복사된다.
3. MAR의 주소를 **주소 버스**로 보내고, **메모리 읽기 신호**(제어 버스)를 함께 보낸다.
4. 메모리가 1000번지의 값을 **데이터 버스**를 통해 **MBR**로 전달한다.
5. 이때 **PC는 1 증가**한다(`1000` → `1001`). → 프로그램을 **순차적으로** 실행할 수 있는 원리.
6. MBR의 명령어가 **IR**로 복사되고, 제어장치가 이를 해석한다.

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/434aa43c-386e-4e98-9b27-2262774ca1dc" />


> **순차적 흐름이 끊기는 경우** — PC는 보통 1씩 증가하지만, JUMP·조건 점프·CALL·RETURN처럼 특정 메모리 주소로 실행 흐름을 옮기는 명령어가 실행되면 다음 번지가 아닌 명시된 주소로 점프한다. 또 인터럽트가 발생해도 순차적 흐름이 끊긴다. (인터럽트는 이후 학습)

특정 레지스터(스택 포인터, 베이스 레지스터)를 이용하는 주소 지정 방식을 아래에서 다룬다.

### 스택 주소 지정 방식

**스택 포인터**는 **스택의 꼭대기를 가리키는** 레지스터로, 스택이 어디까지 차 있는지를 나타낸다. 스택 포인터를 이용한 주소 지정 방식을 **스택 주소 지정 방식**이라고 한다.

> 예) 스택이 아래에서 위로 채워질 때 4번지까지 차 있으면 스택 포인터에 `4번지`가 저장된다. 값이 빠지거나 들어오면 스택 포인터도 그에 맞춰 갱신된다.

> **스택은 어디에 있나?** — 스택은 **메모리 안**에 있다. 메모리에는 스택처럼 쓸 수 있도록 미리 지정된 구역이 있으며, 이를 **스택 영역**이라고 한다. (자세한 내용은 운영체제에서 학습)

### 변위 주소 지정 방식

**변위 주소 지정 방식**은 **오퍼랜드 필드의 값(변위)**과 **특정 레지스터의 값**을 더해 유효 주소를 얻는 방식이다. 따라서 명령어에는 연산 코드, 변위(오퍼랜드), 그리고 더할 레지스터가 함께 명시된다.

대표적으로 두 가지가 있다.

#### 상대 주소 지정 방식

오퍼랜드 필드의 값과 **프로그램 카운터(PC)**의 값을 더해 유효 주소를 얻는다.

> PC에는 다음에 읽어 들일 명령어 주소가 담긴다.
> - `-3` → 다음 명령어 주소에서 3번지 **이전**의 명령어에 접근
> - `+3` → 다음 명령어 주소에서 3번지 **이후**의 명령어에 접근

#### 베이스 레지스터 주소 지정 방식

오퍼랜드 필드의 값과 **베이스 레지스터**의 값을 더해 유효 주소를 얻는다. 베이스 레지스터는 **기준 주소**(프로그램의 시작점 등) 역할을 하고, 오퍼랜드 필드에는 그 기준으로부터 **얼마나 떨어져 있는지**가 담긴다.

> 예) 베이스 레지스터 `1000`, 오퍼랜드 `100` → 1000번지를 기준으로 100번지 떨어진 곳(1100번지)에 접근.

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/f070a625-24d0-4f06-9041-54cb2d2fa84c" />


> **참고** — 베이스 레지스터는 6장(메모리 주소 변환)에서도 등장한다. 메모리에 실제로 저장된 주소와, 프로그램·CPU가 인식하는 주소는 서로 다르다. 이 내용은 이후 주소 변환의 빌드업이 된다.

### CPU 내 레지스터 및 주소 지정 방식 정리

| 레지스터 | 핵심 역할 |
| --- | --- |
| 프로그램 카운터(PC) | 다음에 실행할 명령어 주소 |
| 명령어 레지스터(IR) | 해석할 명령어 |
| 메모리 주소 레지스터(MAR) | 메모리 주소 (주소 버스 경유) |
| 메모리 버퍼 레지스터(MBR) | 메모리와 주고받는 값 (데이터 버스 경유) |
| 플래그 레지스터 | 연산 결과·CPU 상태 부가 정보 |
| 범용 레지스터 | 무엇이든 자유롭게 저장 |
| 스택 포인터 | 스택의 꼭대기 주소 → **스택 주소 지정 방식** |
| 베이스 레지스터 | 기준 주소 → **변위(베이스 레지스터) 주소 지정 방식** |

| 주소 지정 방식 | 유효 주소 계산 |
| --- | --- |
| 스택 주소 지정 | 스택 포인터가 가리키는 스택 꼭대기 |
| 상대 주소 지정 | 오퍼랜드(변위) + 프로그램 카운터 |
| 베이스 레지스터 주소 지정 | 오퍼랜드(변위) + 베이스 레지스터 |
