**최민경**

1-1. 컴파일 언어와 인터프리터 언어의 차이점을 설명하고, 각각의 장단점을 말해주세요.
<details>
  <summary>정답</summary>
  컴파일 언어는 소스 코드 전체를 한 번에 저급 언어로 변환하여 목적 코드를 생성합니다. 초기 컴파일 시간은 걸리지만 실행 속도가 매우 빠르다는 장점이 있습니다. 반면, 인터프리터 언어는 코드를 한 줄씩 해석하며 즉시 실행하므로 별도의 변환 시간을 기다릴 필요가 없고 디버깅이 편리하지만, 실행 속도는 컴파일 언어에 비해 상대적으로 느립니다.
</details>

1-2. 자바나 파이썬 같은 언어들이 컴파일과 인터프리트를 동시에 지원하는 이유는 무엇일까요?
<details>
  <summary>정답</summary>
  자바나 파이썬이 두 방식을 혼용하는 핵심 이유는 코드의 유연한 생산성을 유지하면서도 실행 효율을 극대화하기 위함입니다.
먼저, 소스 코드를 중간 단계인 바이트코드로 미리 컴파일함으로써 얻는 이점이 큽니다. 이 과정을 통해 실행 전 기본적인 문법 검사를 마칠 수 있고, 특정 운영체제에 종속되지 않는 독립성을 확보하게 됩니다. 파이썬이 내부적으로 .pyc 파일을 생성하여 활용하는 것도 이러한 효율성 때문입니다.
인터프리터 방식은 코드를 한 줄씩 해석하므로 실행 속도가 상대적으로 느리다는 단점이 있습니다. 이를 해결하기 위해 도입된 것이 JIT 컴파일러입니다. JIT 컴파일러는 프로그램 실행 중에 인터프리터가 반복적으로 호출되는 이른바 핫스팟 코드를 실시간으로 감지합니다. 그리고 해당 부분만을 기계어로 직접 컴파일하여 캐시에 저장해 둡니다.
결과적으로 자주 사용하는 코드는 재해석 과정 없이 기계어로 즉시 실행되므로, 프로그램이 돌아가면 갈수록 실행 속도가 컴파일 언어 수준으로 최적화됩니다.
</details>

2-1. 명령어 오퍼랜드 필드에 실제 데이터가 아닌 '주소'를 담는 이유가 무엇인가요?
<details>
  <summary>정답</summary>
  명령어의 전체 길이는 CPU 설계에 따라 제한되어 있기 때문입니다. 만약 16비트 명령어에서 연산 코드가 4비트를 차지한다면, 남은 12비트로는 표현할 수 있는 데이터 크기가 매우 작아집니다. 하지만 그 자리에 메모리나 레지스터의 주소를 넣으면 해당 저장 공간의 전체 크기만큼 넓은 범위의 데이터를 활용할 수 있어 훨씬 효율적입니다.
</details>

2-2. cpu 외부에 있는 메모리 접근보다 cpu 내부 레지스터에 접근하는게 더 빠른 이유는 무엇인가요?
<details>
  <summary>정답</summary>
  먼저 물리적 거리가 짧습니다. 레지스터는 CPU 내부 연산 장치에 붙어 있어 데이터 이동 거리가 짧지만, 메모리는 외부 버스를 거쳐야 하므로 물리적 지연이 발생합니다.또한 레지스터는 제어 로직이 단순합니다. 메모리는 주소 변환 및 복잡한 제어 단계를 거쳐야 하지만, 레지스터는 명령어에 직접 매핑되어 있어 별도의 처리 과정 없이 즉시 접근 가능합니다.
</details>

1-2. 리눅스에서 Makefile을 이용하여 컴파일과 링크 과정을 자동화할 때 사용하는 명령어는 무엇인가요?
<details>
  <summary>정답</summary>
  make → Makefile에 정의된 규칙을 기반으로 변경된 파일만 다시 빌드하여 컴파일과 링크를 자동으로 수행
</details>

  컴파일러 → 전체 코드를 한 번에 번역하여 실행<br> 인터프리터 → 한 줄씩 번역하며 실행
</details>

  링크 → 여러 오브젝트 파일과 라이브러리를 결합하여 하나의 실행파일을 생성
</details>
