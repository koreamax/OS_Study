# Chapter 10. 프로세스와 스레드

실행 중인 프로그램을 **프로세스**라 한다. 보조기억장치에 저장된 데이터 덩어리였던 프로그램은 메모리에 적재되어 실행되는 순간 프로세스가 된다(= 프로세스 생성). 이 장에서는 프로세스가 무엇이고, 운영체제가 이를 어떻게 관리하는지 살펴본다.

## 10-1 프로세스 개요

### 프로세스 확인

실행 중인 프로세스는 직접 확인할 수 있다. 윈도우는 **작업 관리자**, macOS·리눅스는 **`ps` 명령어**로 볼 수 있다.

> **포그라운드 / 백그라운드 프로세스**
> - **포그라운드 프로세스** — 사용자와 상호작용하며 실행되는 프로세스
> - **백그라운드 프로세스** — 사용자와 상호작용하지 않고 묵묵히 정해진 일만 수행하는 프로세스. 그중 사용자와 직접 상호작용하지 않는 것을 **데몬(daemon)** 또는 **서비스(service)**라고 부른다. (윈도우 작업 관리자의 '서비스' 탭에서 확인 가능)

### 프로세스 제어 블록 (PCB)

모든 프로세스는 실행을 위해 CPU를 필요로 하지만, CPU 자원은 한정되어 있다. 그래서 프로세스들은 **돌아가면서** 자신에게 할당된 한정된 시간만큼만 CPU를 이용하고, 차례가 끝나면 다음 프로세스에 양보한다.

> **타이머 인터럽트(timer interrupt / 타임아웃 인터럽트)** — "네 차례가 끝났다"를 알리는 인터럽트. 클럭 신호를 발생시키는 장치가 주기적으로 발생시키는 하드웨어 인터럽트의 한 종류다.

운영체제는 빠르게 번갈아 실행되는 프로세스들을 관리하기 위해 **프로세스 제어 블록(PCB, Process Control Block)**이라는 자료구조를 사용한다.

- PCB는 프로세스에 대한 정보를 담은, 상품에 달린 **태그(꼬리표)** 같은 자료구조다.
- 프로세스가 생성되면 **커널 영역**에 PCB가 만들어지고, 프로세스가 종료되면 폐기된다.

PCB에 담기는 대표적인 정보는 다음과 같다. (운영체제마다 차이가 있다.)

| 정보 | 설명 |
| --- | --- |
| **프로세스 ID (PID)** | 프로세스를 식별하는 고유 번호 (학번·사번 같은 존재) |
| **레지스터 값** | 프로세스가 실행 중 사용한 프로그램 카운터·스택 포인터 등 레지스터의 중간값. 차례가 다시 왔을 때 실행을 재개하기 위해 저장 |
| **프로세스 상태** | 생성·준비·실행·대기·종료 등 현재 상태 |
| **CPU 스케줄링 정보** | 언제, 어떤 순서로 CPU를 할당받을지에 대한 정보 |
| **메모리 정보** | 프로세스가 저장된 주소 정보 (베이스·한계 레지스터 값, 특히 **페이지 테이블** 정보) |
| **사용한 파일·입출력장치 정보** | 어떤 입출력장치가 할당되었는지, 어떤 파일을 열었는지 |

> **레지스터 값을 PCB에 저장하는 이유** — 프로세스는 자기 차례가 끝나면 실행을 중단하고 기다린다. 이때 지금까지 CPU에서 쓰던 레지스터 값(다음 실행할 주소, 연산 중간값 등)을 저장해 두어야, 다시 차례가 왔을 때 그 값을 복원해 실행을 **재개**할 수 있다.

### 문맥 교환 (Context Switch)

한 프로세스(A)가 실행되다 다른 프로세스(B)로 실행 순서가 넘어갈 때 거치는 과정이다.

**문맥(context)** 은 다음 차례가 왔을 때 실행을 재개하기 위한 중간 정보(프로그램 카운터를 비롯한 레지스터 값, 메모리 정보, 열었던 파일, 사용한 입출력장치 등)를 말한다.

**문맥 교환**은 다음 두 단계로 이루어진다.

1. 기존 실행 중이던 프로세스 A의 문맥을 A의 **PCB에 백업**한다.
2. 이어 실행할 프로세스 B의 문맥을 B의 **PCB에서 복구**한다.

이렇게 하면 실행 중인 프로세스가 자연스럽게 A → B로 바뀐다. 문맥 교환은 수많은 프로세스가 끊임없이 빠르게 번갈아 실행될 수 있는 원리다.

### 프로세스 메모리 구조

PCB가 커널 영역에 저장되는 반면, 프로세스 자체는 **사용자 영역**에 크게 네 영역으로 나뉘어 저장된다.

| 영역 | 저장 내용 |
| --- | --- |
| **코드 영역(텍스트 영역)** | 기계어 명령어. 바뀔 일이 없어 **읽기 전용(read-only)** |
| **데이터 영역** | 프로그램 실행 내내 유지할 데이터 (예: **전역 변수**) |
| **힙 영역(heap)** | 프로그래머가 직접 할당하는 공간 |
| **스택 영역(stack)** | 일시적으로 저장할 데이터 (예: 매개변수, 지역 변수) |

**정적 할당 영역 vs 동적 할당 영역**

- **코드 영역 + 데이터 영역**: 크기가 고정되어 있어 **정적 할당 영역**이라 한다.
- **힙 영역 + 스택 영역**: 실행 과정에서 크기가 변할 수 있어 **동적 할당 영역**이라 한다.

> - **힙 영역**은 낮은 주소 → 높은 주소로, **스택 영역**은 높은 주소 → 낮은 주소로 할당된다. 서로 반대 방향으로 차올라 주소 충돌을 방지한다.
> - **메모리 누수(memory leak)** — 힙 영역에 할당한 공간을 반환하지 않으면 계속 메모리를 차지해 낭비가 발생하는 문제. (요즘 언어는 **가비지 컬렉션**으로 자동 반환하지만, C처럼 이 기능이 없는 언어는 직접 반환해야 한다.)

## 10-2 프로세스 상태와 계층 구조

### 프로세스 상태

모든 프로세스는 저마다의 상태를 가지며, 운영체제는 이를 PCB에 기록해 관리한다. 대부분의 운영체제가 공통으로 쓰는 상태는 다섯 가지다.

| 상태 | 설명 |
| --- | --- |
| **생성 상태(new)** | 이제 막 메모리에 적재되어 PCB를 할당받은 상태 |
| **준비 상태(ready)** | 당장 CPU를 할당받아 실행할 수 있지만, 아직 자기 차례가 아니라 기다리는 상태 |
| **실행 상태(running)** | CPU를 할당받아 실행 중인 상태 |
| **대기 상태(blocked)** | 실행 중 입출력 작업을 만나, 그 작업이 끝날 때까지 기다리는 상태 |
| **종료 상태(terminated)** | 프로세스가 종료된 상태. 메모리 영역을 정리하고 PCB를 폐기 |

**상태 전이(프로세스 상태 다이어그램)**

- 생성 → **준비**: PCB를 발급받고 실행 준비가 되면 준비 상태로.
- 준비 → **실행**: 자기 차례가 되면 실행 상태로. (이 전환을 **디스패치(dispatch)**라 함)
- 실행 → **준비**: 할당된 시간을 다 쓰면(**타이머 인터럽트**) 다시 준비 상태로.
- 실행 → **대기**: 입출력 작업을 만나면 대기 상태로.
- 대기 → **준비**: 입출력이 완료되면(입출력 완료 인터럽트) 다시 준비 상태로.
- 실행 → **종료**: 실행이 끝나면 종료 상태로.

### 프로세스 계층 구조

대부분의 운영체제는 프로세스를 **계층적으로** 관리한다. (단, 윈도우는 예외적으로 계층적으로 관리하지 않는다. 그러나 개발자가 많이 쓰는 리눅스·macOS·유닉스는 계층 구조를 따르므로 반드시 알아야 한다.)

- 프로세스는 실행 중 **시스템 호출**을 통해 다른 프로세스를 생성할 수 있다.
- 새 프로세스를 생성한 쪽이 **부모 프로세스(parent process)**, 생성된 쪽이 **자식 프로세스(child process)**다.
- 부모와 자식은 별개의 프로세스라 각자 다른 **PID**를 가진다. (같은 프로그램을 두 번 실행해도 PID가 다른 별개의 프로세스다.) 일부 운영체제는 자식의 PCB에 부모의 PID인 **PPID(Parent PID)**를 명시한다.
- 자식 프로세스가 또 자식을 낳는 식으로 **계층(트리) 구조**를 형성한다.

> **최초 프로세스** — 컴퓨터 부팅 시 실행되는 최초의 프로세스가 모든 프로세스의 뿌리가 된다. `pstree` 명령어로 계층 구조를 트리 형태로 볼 수 있다. 리눅스는 **systemd**(옛날엔 init), macOS는 **launchd**가 최초 프로세스 역할을 한다. (데몬·서비스도 이 최초 프로세스의 자식 프로세스로 볼 수 있다.)

### 프로세스 생성 기법 (fork-exec)

부모 프로세스가 자식 프로세스를 만들고, 자식이 자신만의 코드를 실행하는 과정은 **복제**와 **옷 갈아입기**로 비유할 수 있다. (윈도우와는 관련 없으나, 많은 운영체제에서 사용하는 방식이다.)

| 시스템 호출 | 비유 | 동작 |
| --- | --- | --- |
| **fork** | 복제 | 부모 프로세스가 자신의 **복사본**을 자식 프로세스로 생성. 부모의 자원(메모리 내용, 열린 파일 목록 등)이 자식에 상속됨 (단, PID와 저장된 메모리 위치는 다름) |
| **exec** | 옷 갈아입기 | 자식 프로세스가 자신의 메모리 공간을 **새로운 프로그램으로 덮어씀**. 코드·데이터 영역이 새 프로그램으로 바뀌고 나머지 영역은 초기화됨 |

> 예) 사용자가 bash 셸에서 `ls`를 입력하면, bash(부모)가 **fork**로 자신의 복제본(자식)을 만들고, 그 자식이 **exec**로 `ls` 프로그램으로 옷을 갈아입어 실행된다. bash는 그대로 유지된다.

부모 프로세스가 자식을 실행하며 계층 구조를 이루는 과정은 이 **fork-exec가 반복**되는 과정으로 볼 수 있다. (자식이 fork 이후 exec를 하지 않으면, 부모와 똑같은 일을 하는 프로세스가 병행 실행된다.)
