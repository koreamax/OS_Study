# Chapter 08. 입출력장치

컴퓨터 4대 핵심 부품의 마지막인 **입출력장치**를 다룬다. 다양한 입출력장치가 컴퓨터 내부와 어떻게 소통하는지(장치 컨트롤러·장치 드라이버), 그리고 실제로 정보를 주고받는 방법(프로그램 입출력·인터럽트 기반 입출력·DMA)을 살펴본다. (여기서 입출력장치에는 하드 디스크, SSD, USB 메모리 같은 보조기억장치도 포함된다.)

## 08-1 장치 컨트롤러와 장치 드라이버

### 입출력장치가 다루기 까다로운 이유

입출력장치는 CPU·메모리보다 다루기 까다롭다. 크게 두 가지 이유가 있다.

1. **종류가 너무 다양하다** — 마우스, 키보드, 모니터, 프린터 등 종류도 많고, 같은 종류라도 회사·제품마다 모양과 데이터 전송 형식이 다르다. 그래서 컴퓨터 내부와 정보를 주고받는 방식을 **규격화하기 어렵다.**
2. **전송률 차이가 크다** — 일반적으로 CPU·메모리의 데이터 **전송률은 높지만**, 입출력장치의 전송률은 **낮다.** (전송률은 데이터를 얼마나 빨리 교환할 수 있는지를 나타내는 지표다.) 이 차이 때문에 직접 소통하기 어렵다.

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/d4dfff2a-b29f-49cf-9a0f-bc7b02974f86" />


이러한 이유로 등장한 것이 **장치 컨트롤러**다.

### 장치 컨트롤러

**장치 컨트롤러(device controller)**는 입출력 제어기, 입출력 모듈이라고도 하는 **하드웨어**다. 모든 입출력장치는 하나 이상의 장치 컨트롤러와 연결되며(컨트롤러 하나에 여러 입출력장치가 연결될 수도 있음), 입출력장치는 이 장치 컨트롤러를 통해 컴퓨터 내부와 정보를 주고받는다.

#### 장치 컨트롤러의 역할

| 역할 | 설명 |
| --- | --- |
| **통신 중개** | CPU와 입출력장치 간 통신을 중계하는 일종의 번역가 역할. 규격화하기 어렵다는 문제를 완화 |
| **오류 검출** | 연결된 입출력장치에 문제가 없는지 검출 |
| **데이터 버퍼링** | 전송률이 높은 장치와 낮은 장치 사이의 데이터를 **버퍼**(임시 저장 공간)에 모았다가 보내, 전송률 차이를 완화 |

#### 장치 컨트롤러의 구조

장치 컨트롤러는 버스에 연결되어 정보를 주고받으며, 주고받는 정보는 세 종류의 레지스터에 담긴다.

| 레지스터 | 담기는 정보 |
| --- | --- |
| **데이터 레지스터** | CPU와 입출력장치가 주고받을 **데이터**(버퍼 역할). 최근에는 데이터가 많아져 RAM을 쓰기도 함 |
| **상태 레지스터** | 입출력장치의 **상태**(작업 준비 완료 여부, 작업 중 여부, 완료 여부, 오류 여부 등) |
| **제어 레지스터** | 입출력장치가 수행할 **제어 정보**(동작 명령) |

> 상태 레지스터와 제어 레지스터는 하나의 **상태/제어 레지스터**로 합쳐 쓰이기도 한다.

### 장치 드라이버

**장치 드라이버(device driver)**는 장치 컨트롤러의 동작을 감지하고 제어하는 **프로그램**이다.

- 장치 컨트롤러가 하드웨어적인 통로라면, 장치 드라이버는 **소프트웨어적인 통로**다.
- 장치 드라이버가 설치되어 있으면, 회사·규격에 관계없이 컴퓨터가 그 장치를 인식하고 동작시킬 수 있다. 설치되어 있지 않으면 그 장치를 쓸 수 없다.
- 새 입출력장치가 인식되지 않을 때 가장 먼저 확인할 것이 장치 드라이버다.

> 장치 드라이버를 인식하고 실행시키는 주체는 **운영체제**다. (운영체제는 다음 장부터 자세히 다룬다.)

## 08-2 다양한 입출력 방법

입출력 방식에는 대표적으로 **프로그램 입출력**, **인터럽트 기반 입출력**, **DMA 입출력** 세 가지가 있다.

### 프로그램 입출력

프로그램 속 명령어로 (입출력장치에 연결된) **장치 컨트롤러를 제어**하는 방법이다. CPU가 장치 컨트롤러의 레지스터 값을 **읽고 씀**으로써 입출력이 이루어진다.

> 예) 메모리 정보를 하드 디스크에 백업(쓰기)하는 과정
> 1. CPU가 하드 디스크 컨트롤러의 **제어 레지스터**에 쓰기 명령을 보낸다.
> 2. 컨트롤러가 하드 디스크 상태를 확인해, 준비되면 **상태 레지스터**에 "준비 완료"를 표시한다.
> 3. CPU가 상태 레지스터를 주기적으로 읽어 준비 여부를 확인한다.
> 4. 준비되면 백업할 정보를 **데이터 레지스터**에 쓴다. (백업이 끝날 때까지 반복)

#### 입출력장치의 주소 — 메모리 맵 입출력 vs 고립형 입출력

CPU가 수많은 장치 컨트롤러의 레지스터(입출력장치의 주소)를 어떻게 인식하느냐에 따라 두 방식이 있다.

| 방식 | 설명 | 특징 |
| --- | --- | --- |
| **메모리 맵 입출력(Memory-mapped I/O)** | 메모리 접근 주소 공간과 입출력장치 접근 주소 공간을 **하나의 주소 공간**으로 간주 | 메모리 접근 명령어와 입출력 명령어가 **동일**(별도 입출력 명령어 불필요). 단, **메모리 주소 공간이 축소**됨 |
| **고립형 입출력(Isolated I/O)** | 메모리 주소 공간과 입출력장치 주소 공간을 **분리** | 메모리 주소 공간이 축소되지 않음. 단, **입출력 전용 명령어**(예: x86의 `in`, `out`)가 필요 |

> 고립형 입출력은 제어 버스에 메모리 읽기/쓰기 선과 별도로 **입출력장치 읽기/쓰기 선**이 있어, 어느 선을 활성화하느냐로 메모리와 입출력장치를 구분한다.

### 인터럽트 기반 입출력

프로그램 입출력에서는 CPU가 상태 레지스터를 **주기적으로 읽으며**("준비됐니? 아니네"의 반복) 확인해야 한다. 이를 **폴링(polling)**이라 한다. 인터럽트 기반 입출력은 이 낭비를 막는다.

- 장치 컨트롤러가 입출력 작업을 끝내면 CPU에 **인터럽트 요청 신호**를 보낸다. CPU는 요청을 받았을 때만 처리하면 되므로 CPU 사이클을 효율적으로 쓸 수 있다.
- (하드웨어 인터럽트는 정확히는 입출력장치가 아니라 **장치 컨트롤러**가 발생시킨다.)

#### 다중 인터럽트 처리

여러 입출력장치가 동시에 인터럽트 요청을 보낼 수 있다. 처리 방식은 두 가지다.

| 방식 | 설명 |
| --- | --- |
| **순차 처리** | 인터럽트 비트를 비활성화한 채 발생 순서대로 처리. 단, 모든 인터럽트를 순차 처리할 수는 없음 |
| **우선순위 반영** | 우선순위가 높은 인터럽트를 먼저 처리. ISR 실행 중에도 더 높은 우선순위 인터럽트가 오면 중단하고 그쪽을 먼저 처리 |

> 우선순위를 반영한 다중 인터럽트 처리에는 보통 **PIC(Programmable Interrupt Controller)**라는 하드웨어를 쓴다. PIC는 여러 장치 컨트롤러에 연결되어, 들어온 하드웨어 인터럽트들의 우선순위를 판단해 CPU에 알려준다. (단, 하드웨어 고장·정전 같은 **NMI**(Non-Maskable Interrupt)의 우선순위는 판단하지 않으며, NMI는 별도 핀으로 CPU에 바로 전달된다.)

### DMA 입출력

프로그램 입출력과 인터럽트 기반 입출력은 공통적으로, 입출력장치와 메모리 간의 데이터 이동을 **CPU가 주도**하고 이동되는 데이터가 **반드시 CPU를 거친다.** 대용량 데이터 이동 시 CPU 부담이 커진다.

**DMA(Direct Memory Access)**는 **CPU를 거치지 않고** 입출력장치가 메모리에 직접 접근하는 입출력 방식이다. 이를 위해 **DMA 컨트롤러**라는 하드웨어가 필요하다.

#### DMA 입출력 과정

1. CPU가 **DMA 컨트롤러**에 입출력 작업을 명령한다. (어떤 장치에, 메모리 몇 번지에, 어떤 연산을 할지)
2. DMA 컨트롤러가 CPU를 **대행**해 장치 컨트롤러와 상호작용하며, 필요할 때 **메모리에 직접 접근**해 읽고 쓴다. (이 동안 CPU는 다른 작업 수행)
3. 입출력 작업이 끝나면 DMA 컨트롤러가 **인터럽트**로 CPU에 완료를 알린다.

→ CPU는 입출력 작업의 **시작과 끝**에만 관여하면 된다.

<img width="1448" height="1086" alt="image" src="https://github.com/user-attachments/assets/659d26eb-581c-4182-848b-6b0a763964ea" />


> **사이클 스틸링** — 시스템 버스는 공용 자원이라 CPU와 DMA 컨트롤러가 동시에 쓸 수 없다. 그래서 DMA 컨트롤러는 CPU가 시스템 버스를 쓰지 않을 때 조금씩 빌려 쓴다. 이를 **사이클 스틸링(cycle stealing)**이라 한다.

#### 입출력 버스

장치 컨트롤러가 시스템 버스에 직접 연결되면, DMA 과정에서 시스템 버스를 **불필요하게 두 번**(읽기·쓰기) 점유하게 된다. 이를 막기 위해 장치 컨트롤러는 **입출력 버스(I/O bus)**라는 전용 버스에 연결된다. 입출력 버스는 다시 시스템 버스에 연결된다.

- 대표적인 입출력 버스: **PCI 버스, PCI Express(PCIe) 버스**. (컴퓨터의 슬롯이 입출력 버스와 연결된다.)

> **입출력 채널(입출력 전용 프로세서)** — 최근 DMA는 더 발전해, 입출력 작업 명령어 실행까지 담당하는 입출력 전용 프로세서를 두기도 한다. (예: "듀얼코어 CPU 탑재 프린터" — 입출력장치가 RAM이나 전용 CPU를 탑재.)

### 입출력 방법 정리

| 방식 | 데이터 이동 주체 | 특징 |
| --- | --- | --- |
| **프로그램 입출력** | CPU (폴링) | CPU가 상태 레지스터를 주기적으로 확인. 메모리 맵 / 고립형으로 나뉨 |
| **인터럽트 기반 입출력** | CPU (인터럽트) | 작업 완료 시 인터럽트로 알림. CPU 사이클 효율↑. PIC로 우선순위 처리 |
| **DMA 입출력** | DMA 컨트롤러 | CPU를 거치지 않고 메모리에 직접 접근. CPU는 시작·끝만 관여 |
